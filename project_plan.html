1. The Pitch

What it is: A traffic congestion decision-support platform that ingests road sensor speed data, uses GPU-accelerated (cuDF) processing to compute corridor-level congestion metrics at a scale/speed pandas can't match, forecasts near-term congestion buildup per corridor, and outputs a ranked, actionable list: which corridors need intervention (signal retiming / rerouting), how urgent, and what impact is modeled. A Gemini-powered natural language layer sits on top so a city ops person can ask "which corridors need attention this evening" and get a direct answer.

Built on: METR-LA / PEMS-BAY (real Caltrans sensor network data, industry-standard traffic ML benchmark, 200+ sensors, 5-min granularity). Framed explicitly as a city-agnostic architecture — deployable to any city with a comparable sensor/GPS feed.

2. Why This Beats The Obvious Idea

  • Agri-price/mandi arbitrage ideas are the default pitch at every India-based Google Cloud hackathon — you already correctly killed that one for saturation risk.
  • Hyderabad-specific traffic data does not exist in usable form (checked: GHMC/HMDA open data portals only have land-use, boundaries, waterbodies, air quality — no live/historical vehicle flow or signal data). Don't fake a local dataset and get caught in Q&A — use the honest "built on X, deployable anywhere" framing instead.
  • This idea forces a legitimate O(n²)-ish aggregation (sensor-pair / corridor-pair congestion propagation across thousands of time windows) — a real place where cuDF vs pandas shows a genuine, provable speedup. That benchmark is your single strongest differentiator; most teams will claim "GPU-accelerated" and never show a number.

5. The Acceleration Benchmark — Non-Negotiable

This is the single highest-leverage thing you build. Do not skip it, do not fake it.

  • Pick ONE transform that is genuinely heavy: e.g. computing rolling-window congestion index + pairwise corridor delta across all sensors × all timestamps.
  • Run it twice: once in plain pandas, once in cudf.pandas (drop-in accelerator — literally %load_ext cudf.pandas before importing pandas, zero code rewrite needed if you're on Colab with GPU runtime).
  • Log wall-clock time for both. Screenshot or print the numbers. This becomes your opening demo slide — a real, numeric, undeniable "Nx faster" claim beats every team that just says "GPU-accelerated" with no evidence.
  • If cudf.pandas import fails / no GPU runtime available in time: fallback to full import cudf API on a Colab GPU runtime (free tier T4 is enough for this data size) — do this test FIRST, before building anything else, since your whole differentiator depends on it working.

6. Modeling & Decision Output

  • Don't overbuild. A gradient-boosted or even simple rolling-average-based short-horizon forecast per sensor/corridor is enough — judges care about the decision-support framing, not SOTA modeling.
  • Score formula (keep it simple and explainable): 
    urgency_score = predicted_congestion_increase × corridor_traffic_volume_weight
  • Output row example: Corridor: US-101 N @ Sensor 717447 | Urgency: 8.2/10 | Predicted congestion peak: 18:40 | Recommended: extend green phase +15s / reroute advisory | Modeled impact: -12% avg wait
  • Rank all corridors by urgency_score, surface top N in the dashboard — this ranked list IS your decision-support output.

8. Gemini NL Layer

  • Simplest viable version: Gemini API call with the ranked corridor table serialized as context, answering questions like "which corridors need attention this evening" or "why is corridor X flagged."
  • You already have ADK/multi-agent experience from GenAI Academy Track 1 — reuse that pattern if time allows, but a single well-prompted Gemini call over the table is sufficient for the demo. Don't over-engineer the agent framework under time pressure.

9. Dashboard

  • If Looker setup is fast for you — use it, it's a named Google Cloud product and scores well.
  • If Looker eats time: a simple Streamlit or even a static HTML page rendering the ranked table + a couple of charts (matplotlib/plotly) is a legitimate, honest fallback. A working simple dashboard beats a half-configured Looker instance during a live demo.

11. Antigravity Task Breakdown

Feed these as discrete, ground-truth-locked tasks — same pattern as your ATIRP PROJECT_SPEC.md approach.

  • Task 1: Data ingestion script — download, validate schema, push to Cloud Storage.
  • Task 2: Transform module — pandas version AND cudf.pandas version of the identical function, with timing decorators on both, output a comparison log/JSON.
  • Task 3: BigQuery loader — schema definition, load cleaned aggregates.
  • Task 4: Scoring module — urgency_score formula, ranked table generation.
  • Task 5: Forecast module — simple per-corridor short-horizon prediction feeding into Task 4.
  • Task 6: Gemini NL query function — takes ranked table + user question, returns answer.
  • Task 7: Dashboard render — ranked table view + benchmark bar chart (pandas vs cudf.pandas time) as the hero visual.

Run overnight/parallel where possible — Tasks 1–3 are sequential and gate everything, but 5, 6, 7 can be parallelized once Task 4's table shape is locked.
